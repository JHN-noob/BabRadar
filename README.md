# babradar

## 프로젝트 개요

`babradar`는 주변 음식점을 지도에서 탐색하고, 실제 후보 데이터만 기반으로 추천을 제공하는 모바일 서비스입니다.

- `mobile/`: Expo + React Native 클라이언트
- `server/`: Next.js 백엔드

MVP 목표는 다음과 같습니다.

- 현재 위치 권한을 받고 지도에 반영
- 주변 음식점 마커와 리스트를 함께 표시
- 음식점 상세 화면 제공
- 이메일 로그인 및 즐겨찾기 제공
- 실제 후보군만 기반으로 1인 식사 추천 제공

## 현재 상태

- `mobile/`은 제품 기준 탭/상세/인증 라우트 골격과 지도형 홈 UI 목업까지 정리된 상태입니다.
- `server/`는 아직 Next.js 스타터 템플릿 단계입니다.
- 루트 문서와 초기 파이프라인 문서는 생성되어 있습니다.
- 모바일 앱 식별자는 폴더명에 맞춰 `mobile`로 정리했습니다.

## 기술 스택

- 모바일: Expo, React Native, TypeScript, Expo Router
- 백엔드: Next.js App Router, TypeScript
- 인증/DB: Supabase Auth, Supabase DB
- 외부 데이터: Kakao Local API
- AI 추천: OpenAI API

## 실행 방법

모바일 앱:

```bash
cd mobile
npm run start
```

백엔드:

```bash
cd server
npm run dev
```

주의:

- 민감한 키는 모두 서버에서만 사용합니다.
- 기능 구현은 작은 단계로 나누고, 단계마다 실행 확인 후 다음 작업으로 넘어갑니다.

## 환경 변수 가이드

서버 전용 비밀값:

- `KAKAO_REST_API_KEY`
- `OPENAI_API_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

클라이언트에 노출 가능한 공개값:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `EXPO_PUBLIC_SUPABASE_URL`
- `EXPO_PUBLIC_SUPABASE_ANON_KEY`
- `EXPO_PUBLIC_SERVER_BASE_URL`

원칙:

- Expo 앱에 OpenAI API 키를 넣지 않습니다.
- Expo 앱에 Kakao 비밀키를 넣지 않습니다.
- Expo 앱에 Supabase service role 키를 넣지 않습니다.
- 외부 API의 민감한 호출은 모두 `server/`를 통해 처리합니다.

## 초기 분석

### 제품 목표 요약

지도 중심 UX로 주변 음식점을 빠르게 탐색하고, 로그인 사용자에게 즐겨찾기와 1인 식사 추천까지 제공하는 것이 목표입니다.

### 사용자 흐름

1. 앱 실행
2. Supabase 세션 복원
3. 위치 권한 요청
4. 현재 위치 기준으로 지도 이동
5. 모바일이 서버에 주변 음식점 조회 요청
6. 서버가 Kakao Local API 호출 후 데이터를 정규화
7. 모바일이 마커와 하단 리스트를 동기화해서 표시
8. 사용자가 정렬, 카테고리, 상세 화면을 탐색
9. 로그인 사용자는 즐겨찾기 저장/해제
10. 현재 후보군 기준으로 AI 추천 요청
11. 서버가 실제 후보군만 OpenAI에 전달
12. 추천 결과와 이유를 모바일에 반환

### 화면 구조

- 앱 부트스트랩 / 세션 복원
- 로그인 / 회원가입
- 지도 메인 화면
- 바텀시트 음식점 리스트
- 음식점 상세 화면
- 즐겨찾기 화면
- 마이페이지
- 추천 화면

### 데이터 흐름

1. 모바일이 위치와 필터 상태를 관리
2. 모바일이 좌표, 반경, 카테고리, 정렬 옵션을 서버로 전달
3. 서버가 Kakao Local API를 호출
4. 서버가 응답을 정규화해 모바일에 전달
5. 모바일이 지도, 리스트, 상세 진입 상태를 렌더링
6. Supabase Auth로 로그인 세션을 관리
7. 즐겨찾기는 사용자 기준으로 저장/조회
8. 추천은 현재 조회된 실제 후보군만 서버로 전달
9. 서버가 OpenAI로 추천 사유를 생성
10. 모바일이 추천 카드와 설명을 표시

### 인증 흐름

1. 이메일/비밀번호 로그인 또는 회원가입
2. 모바일에 세션 저장
3. 보호가 필요한 기능은 세션 확인 후 수행
4. 즐겨찾기와 마이페이지는 사용자 ID 기준으로 조회

### 즐겨찾기 흐름

1. 사용자가 리스트 또는 상세에서 즐겨찾기 버튼 탭
2. 모바일이 로그인 상태 확인
3. Supabase에 저장 또는 삭제
4. 리스트, 상세, 즐겨찾기 화면 상태를 동기화

### 음식점 상세 흐름

1. 마커 또는 리스트 아이템 탭
2. 상세 화면으로 이동
3. 실제 응답 데이터만 표시
4. 즐겨찾기 버튼 제공
5. 외부 링크가 있으면 이동 지원

### AI 추천 흐름

1. 모바일이 현재 후보군을 수집
2. 후보군과 추천 주제를 서버에 전달
3. 서버가 실제 조회 데이터인지 검증
4. OpenAI에 1인 식사 기준 추천 요청
5. 서버가 순위와 추천 이유를 구조화해 반환
6. 모바일이 추천 결과를 렌더링

### 주요 리스크와 미확정 사항

- Expo Go에서 지도/바텀시트 조합이 원하는 수준으로 동작하는지
- Kakao API에서 평점/리뷰 수 정렬에 필요한 데이터가 충분한지
- 지도 이동과 리스트 동기화 시 성능 이슈가 없는지
- 상세 화면에 필요한 데이터가 Kakao 응답만으로 충분한지
- 세션 복원과 토큰 갱신 처리에서 엣지 케이스가 없는지
- 추천 결과가 실제 후보 데이터만 사용하도록 안정적으로 제한되는지

### MVP 경계

- Kakao API 우선
- 이메일 로그인 우선
- 즐겨찾기는 저장/삭제/목록만 구현
- 추천은 1인 식사 중심
- 지원되지 않는 평점/리뷰 데이터는 꾸며서 보여주지 않음
- 비밀키는 모바일에 저장하지 않음

## 구현 파이프라인

### 1. 모바일 UI 흐름

1. 스타터 탭 구조를 제품 라우트 구조로 교체
2. 지도 메인 화면과 위치 상태 구성
3. 바텀시트 음식점 리스트 추가
4. 상세 화면과 즐겨찾기 액션 연결

### 2. 모바일 -> 서버 요청 흐름

1. 요청/응답 타입 정의
2. `mobile/lib/`에 API 클라이언트 추가
3. 위치/필터 기반 조회 요청 연결
4. 네트워크 상태와 UI 상태 분리

### 3. 서버 -> 외부 API 흐름

1. `server/src/services/kakao/`에 Kakao 연동 모듈 작성
2. 장소 데이터 정규화 레이어 추가
3. nearby 검색, 추천용 API 라우트 추가
4. 민감한 키는 서버 내부에서만 사용

### 4. Supabase 인증/데이터 흐름

1. 모바일 Supabase 클라이언트 구성
2. 이메일 로그인/회원가입 화면 추가
3. 즐겨찾기 테이블 및 정책 구성
4. 사용자 상태와 즐겨찾기 상태 동기화

### 5. AI 추천 처리 흐름

1. 실제 음식점 후보군 스키마 정의
2. 현재 조회된 후보만 서버에 전달
3. 1인 식사 적합성 중심 프롬프트 구성
4. 과장 없이 짧고 실용적인 추천 이유 반환

## 초기 보안 점검

2026-03-26 기준 현재 저장소 기준으로 간단 점검한 결과입니다.

- 루트, `mobile/`, `server/` 하위에 `.env` 파일은 존재하지 않았습니다.
- 저장소 검색 기준으로 OpenAI, Kakao, Supabase 비밀키가 하드코딩된 흔적은 없었습니다.
- 현재는 서버 API와 인증 로직이 아직 본격 구현 전이라, 입력 검증/RLS/세션 보안 점검은 이후 단계에서 다시 수행해야 합니다.

추가로 확인할 항목:

- 추천 API 요청 크기 제한과 남용 방지
- Supabase RLS 정책 과도 허용 여부
- 서버 라우트 입력 검증
- 민감 정보 로그 출력 여부

- 탭: 지도, 즐겨찾기, 추천, 마이페이지

- 제스처: 높이 임계값과 빠른 플릭 속도에 따라 최소/전체화면으로 스냅, 상태 전환 시 햅틱 반응
- 하단 탭: 휴대폰 제스처 영역을 고려한 safe area 여백 적용
